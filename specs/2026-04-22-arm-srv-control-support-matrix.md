## 1. Overview / Цель
Провести ревизию реально используемых UI-контролов и пользовательских сценариев из проекта `Arm.Srv`, сопоставить их с текущими возможностями AppAutomation и добавить результат в `ControlSupportMatrix.md`.

Итоговая матрица должна явно показать, какие сценарии уже закрываются текущими слоями recorder / headless / FlaUI, а какие требуют дополнительной поддержки.

## 2. Текущее состояние (AS-IS)
- `ControlSupportMatrix.md` сейчас описывает demo/showcase AppAutomation: Avalonia controls, стандартный `DataGrid`, Eremex `DataGrid` через bridge.
- В `Arm.Srv` есть desktop-клиент на Avalonia с большим числом `.axaml` views в `src/Arm.Client`.
- По первичному поиску в `Arm.Srv` используются стандартные Avalonia-контролы, Eremex controls/editors, кастомные `miniControls`, гриды с сохранением layout, server sorting, copy/export/open-on-double-click и поисковые сценарии.
- Текущая матрица не содержит отдельного раздела по потребительскому проекту `Arm.Srv` и не фиксирует gaps по recorder/headless/FlaUI для этих сценариев.

## 3. Проблема
Нет проверяемой сводки по фактическим UI-контролам и сценариям `Arm.Srv`, поэтому нельзя приоритизировать доработки AppAutomation для реального потребительского приложения.

## 4. Цели дизайна
- Разделение ответственности: матрица фиксирует анализ и gaps, без изменения runtime-кода.
- Повторное использование: использовать существующие понятия матрицы (`UiControlType`, support level, recorder/headless/FlaUI).
- Тестируемость: результат проверяется повторяемыми поисковыми командами и ревью измененного markdown.
- Консистентность: сохранить двуязычную структуру `ControlSupportMatrix.md`; добавить русскоязычный потребительский раздел и, при необходимости, краткий английский аналог.
- Обратная совместимость: не менять публичные API, селекторы, тестовые сценарии или поведение приложения.

## 5. Non-Goals (чего НЕ делаем)
- Не добавлять поддержку новых контролов в AppAutomation.
- Не менять XAML, ViewModel, тесты или инфраструктуру `Arm.Srv`.
- Не запускать UI-приложение и не выполнять ручную запись сценариев recorder.
- Не гарантировать полную статическую инвентаризацию каждого отдельного экземпляра контрола; цель - покрыть типы контролов и повторяющиеся сценарии использования.
- Не делать выводы о runtime UIA-дереве без фактического запуска; такие места фиксировать как риск или требующие проверки.

### 6.2 Детальный дизайн

  - именованных controls / automation anchors;

  - "Recommended automation contract for Arm.Srv".

## 7. Бизнес-правила / Алгоритмы
- Один тип контрола может попадать в несколько сценариев, если сценарии требуют разных automation primitives.
- Для Eremex `DataGridControl` отдельно учитывать:
  - базовые grid assertions;
  - выбор строк/ячеек;
  - сортировку по заголовку;
  - load more / lazy loading;
  - copy/export;
  - layout persistence;
  - double click/open card.
- Если статический анализ показывает кастомный wrapper (`miniControls`), в матрице фиксировать wrapper и предполагаемые внутренние primitives, не выдавая предположение за runtime-факт.

## 12. Риски и edge cases
- Статический XAML-анализ может не увидеть controls, созданные динамически в code-behind.
- Eremex controls могут иметь runtime UIA-поведение, которое нельзя надежно подтвердить без запуска приложения.
- Некоторые кастомные `miniControls` могут скрывать несколько базовых controls; в матрице нужно избегать чрезмерно точных утверждений без проверки исходника wrapper'а.
- Большой объем `.axaml` может дать шум; нужно группировать по reusable patterns, а не перечислять каждый элемент.

## 18. Альтернативы и компромиссы
- Вариант: перечислить каждый найденный control instance.
  - Плюсы: максимальная детализация.
  - Минусы: матрица станет шумной и быстро устареет.
  - Почему не выбран: цель задачи - поддержка типов и сценариев, а не полный inventory XAML.
- Вариант: группировать controls по reusable сценариям.
  - Плюсы: читаемо, пригодно для планирования доработок AppAutomation.
  - Минусы: отдельные редкие instances могут быть описаны обобщенно.
  - Почему выбранное решение лучше: оно соответствует назначению матрицы поддержки.
