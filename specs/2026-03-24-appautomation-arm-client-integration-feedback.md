# AppAutomation Integration Feedback For Arm.Client

## Цель документа

- Зафиксировать реальные сложности подключения `AppAutomation` к `Arm.Client`.
- Сохранить историю решений по внедрению UI-тестов.
- Подготовить обратную связь для разработчиков библиотеки и внутренней команды.

## Актуальный статус на 2026-03-24

- `FlaUI`-раннер остаётся основным рабочим раннером для smoke-сценариев.
- `Headless`-раннер теперь тоже выполняет login-сценарии `success/failure` без `skip`.

## Текущее поведение тестового хоста

По умолчанию тесты ожидают реальный тестовый сервер и берут конфигурацию из окружения:

- `ARM_UI_TEST_SERVER_URL`
- `ARM_UI_TEST_LOGIN`
- `ARM_UI_TEST_PASSWORD`

Если логин/пароль не заданы, тестовый хост завершает запуск с явной диагностикой, а не пытается использовать устаревшие хардкоды.

## Что пришлось изменить в приложении

- Вынести reusable bootstrap из `src/Arm.Client.Desktop/Program.cs` в отдельный `AppBootstrap`.
- Добавить `InternalsVisibleTo` для `Arm.AppAutomation.TestHost`.
- Добавить стабильные `AutomationId` в login-shell и main window.
- Усилить защиту от запуска Velopack update-check в неустановленном приложении.

## Что пришлось изменить в тестовой инфраструктуре

- Восстановить source-код тестовых проектов `tests/Arm.UiTests.*` и `tests/Arm.AppAutomation.TestHost`.
- Убрать зависимость authoring-сценариев от backend-specific счётчиков успешных/неуспешных логинов.
- Перевести проверки success/failure на реальные пользовательские сигналы:
  - success: заголовок окна;
  - failure: сохранение login-shell + возможность повторной попытки.
- Упростить тестовый хост до одного server-only контура без локального stub backend.

## Основные сложности интеграции

### 1. Успешный login зависит не от одного endpoint

Для `Arm.Client` недостаточно одного `POST /auth/ldap/login`.
После удачного входа клиент ожидает дополнительные backend-вызовы и SignalR-подключение, иначе UI не переходит в рабочее состояние.

Минимально значимая цепочка:

- `POST /api/arm/auth/ldap/login`
- `POST /api/arm/rules/user`
- `GET /api/arm/users/me`
- `GET /api/arm/users/filters`
- `GET /api/arm/table-layouts/{userId}`
- `POST /api/arm/orders/get`
- `/api/arm/notificationHub`

### 2. Реальный тестовый сервер живой, но тестовые учётные данные со временем протухают

Во время интеграции выяснилось, что сетевой доступ до тестового сервера есть, но заранее известный тестовый пользователь больше не годится для smoke.

Подтверждённый факт:

- 2026-03-24 сервер `http://arm-srv-dev.isolon:5000/api/arm` отвечал на логин `Test1@isolon.ru` кодом `403 Forbidden`.
- Причина из ответа сервера: пользователь удалён.
- 2026-03-24 альтернативный логин `Test@isolon.ru` из старых комментариев в репозитории отвечал кодом `401 Unauthorized`.
- Причина из ответа сервера: неверный логин или пароль.

Практический вывод:

- для server-mode нельзя полагаться на старые хардкоды из репозитория;
- нужен внешний источник актуальных тестовых credentials;
- логин/пароль лучше задавать через переменные окружения, а не через зашитые значения в тестах.

### 3. Без AutomationId внедрение остаётся хрупким

Существующий UI не был изначально подготовлен для automation-first сценариев.
Для первого smoke пришлось вручную добавить `AutomationId` в XAML.

Вывод:

- без заранее продуманной automation-разметки внедрение сильно замедляется;
- динамические dock-pane особенно чувствительны к отсутствию стабильных идентификаторов.

### 4. Dock-based UI дал нестабильный сигнал успешного логина

Изначально успешный тест пытался проверять появление `ordersViewModel` в dock layout.
На практике backend-flow мог уже успешно завершиться, но сам dock-элемент не всегда оказывался стабильной automation-целью.

Итоговое решение:

- финальные проверки идут по пользовательским признакам:
  - обновление заголовка главного окна.

### 5. Для real-server режима понадобился дополнительный UI-сигнал статуса

Когда тесты были отвязаны от внутренних backend-specific счётчиков, исчез и прежний простой способ понять результат логина.
Для стабильного smoke пришлось добавить в UI отдельный automation-доступный статус авторизации.

Почему это оказалось полезно:

- success/failure определяются по состоянию самого клиента, а не по внутреннему тестовому backend;
- тесты не зависят от того, как именно устроена backend-обвязка;
- проверка становится ближе к состоянию самого клиента.
- скрытый automation-маркер больше не является основным бизнес-assert'ом теста, а остаётся только внутренней синхронизацией.

### 6. Headless startup потребовал явного bootstrap hook

Headless/test-host не мог просто "запустить приложение как есть".
Пришлось отдельно инициализировать сервисы через `BeforeLaunchAsync` и вручную создавать `MainWindow`.

Вывод:

- для реальных Avalonia-приложений нужен более явный consumer API для разделения:
  - bootstrap/init;
  - создания окна;
  - подмены config/workspace.

### 7. Avalonia headless ломался на login command из-за thread affinity, но проблема закрыта

После исправления порядка старта headless-сценарии всё равно падали на выполнении логина с ошибкой:

`Call from invalid thread`

Подтверждённая причина:

- проблема была не в backend;
- проблема была не в поиске UI-элементов;
- проблема проявлялась именно в headless execution вокруг Avalonia/ReactiveUI command pipeline;
- отдельный trigger был в том, что `ReactiveCommand` и связанные UI-bound обновления захватывали thread-affinity, а прямой вызов login flow из `Dispatch()` ещё и подвешивал continuation на UI-context headless-раннера.

Итоговое решение:

- `Headless` больше не использует `LoginCommand.Execute()` для automation login flow;
- headless-сессия получает `LoginViewModel` через `Dispatch()`, а сам `AuthorizeForAutomationAsync()` вызывается уже вне UI-thread, чтобы не подвешивать continuation на headless UI-context;
- UI-bound обновления в `LoginViewModel` и `UserRightsManager` маршалятся обратно на главный поток;
- перед удалением временного workspace вызывается `AppServicesProvider.Reset()`, чтобы освободить file watchers от временного `Settings.json`;
- login-сценарии в `Headless` снова зелёные и marker-test о known limitation больше не нужен.

Проверка на 2026-03-24:

- `Arm.UiTests.Headless.exe`: 2 login-сценария passed;
- `Arm.UiTests.FlaUI.exe`: 2 login-сценария passed;
- проблема с `Call from invalid thread` и post-run `UnobservedTaskException` больше не воспроизводится.

### 8. Velopack update check шумел в тестовом запуске

Во время тестового старта приложение пыталось выполнять update-check в окружении, где оно не установлено через Velopack.

Итог:

- добавлена более жёсткая защита;
- update-check теперь пропускается в таком запуске.

### 9. Параллельная сборка раннеров конфликтует по desktop-артефактам

`FlaUI` и `Headless` нельзя бездумно собирать параллельно в одном workspace.
Они делят одни и те же артефакты клиентского приложения, что приводило к блокировкам `dll/pdb`.

Вывод:

- такие проекты лучше собирать последовательно;
- это особенно важно для Avalonia desktop-приложений с weaving/post-processing.

### 10. Ошибки pre-launch hook плохо читаются в TUnit

Когда проблема возникает ещё до старта UI-сессии, например на этапе создания workspace и проверки server credentials,
TUnit в отчёте показывает в основном `TargetInvocationException` из `BeforeTest`-hook.

Практическая проблема:

- корневая причина теряется;
- пользователю сложнее понять, что именно не задано в окружении;
- даже при локальном оборачивании исключения сообщение в стандартном отчёте раннера остаётся слишком общим.

Вывод:

- для реальных UI-раннеров полезен отдельный preflight/diagnostics слой;
- библиотеке/раннеру стоило бы лучше раскрывать inner exception из setup hooks.

### 11. Ложный pre-launch fail из-за неправильного имени env vars

Во время отладки выяснилось, что `AuthorizeFlaUiTests` могут падать ещё до старта UI не только из-за отсутствующих credentials,
но и из-за простой wiring-ошибки в тестовом хосте.

Подтверждённый факт:

- в `ArmAutomationTargetSettings` вместо имён переменных окружения `ARM_UI_TEST_LOGIN` и `ARM_UI_TEST_PASSWORD`
  по ошибке были записаны literal-значения старых credentials;
- из-за этого тестовый процесс игнорировал корректно заданные env vars и всегда считал логин/пароль пустыми;
- после исправления имён env vars `FlaUI` перестал падать в pre-launch и снова начал реально стартовать приложение.

Практический вывод:

- при проблемах "тест падает ещё до запуска" надо проверять не только сами env vars, но и mapping-код, который их читает;
- полезно логировать effective test settings без вывода секрета пароля.

## Что уже подтверждено запуском

### Проверено против реального сервера

- URL тестового сервера доступен.
- Логин `Test1@isolon.ru` на дату 2026-03-24 не подходит для позитивного smoke, потому что сервер возвращает `403 Forbidden` и причину "пользователь удалён".

## Практические рекомендации внутри проекта

- Основной рабочий раннер для smoke/regression: `FlaUI`.
- Реальный test server использовать как основной backend для проверки реального контура.
- Server-only модель держать простой и прозрачной: один backend, один способ запуска.
- `Headless` можно использовать для тех же login smoke-сценариев, но сборку и запуск `Headless`/`FlaUI` лучше держать последовательными.
- Новые экраны сразу размечать `AutomationId`, а не добавлять их постфактум.

## Вопросы и пожелания к разработчикам AppAutomation

- Нужен ли более явный consumer API для bootstrap/init сценариев реальных Avalonia-приложений?
- Можно ли добавить рекомендованный паттерн для isolated workspace/config lifecycle?
- Можно ли улучшить поддержку dock-based UI, где успешное состояние удобнее отслеживать не через стандартные pane/tab-паттерны?
- Можно ли дать built-in helper или guidance для Avalonia headless + ReactiveUI command scenarios, чтобы не приходилось вручную обходить `ReactiveCommand` и UI-thread affinity?
- Можно ли добавить диагностический режим для поиска отсутствующих `AutomationId` в уже существующем приложении?
- Можно ли лучше показывать первопричину ошибок из `BeforeTest`/launch hooks, не сваливаясь в общий `TargetInvocationException`?

## Visual Studio Local Bootstrap Note

На 2026-03-24 в тестовом хосте добавлен fallback-режим для локального запуска из Visual Studio:

- сначала `Arm.UiTests.*` читают `ARM_UI_TEST_SERVER_URL`, `ARM_UI_TEST_LOGIN` и `ARM_UI_TEST_PASSWORD`;
- если они не заданы, читают legacy-переменные `Authentication__LDAP__UserName` и `Authentication__LDAP__Password`;
- если и они не заданы, автоматически разбирают `src/Arm.Srv/Properties/launchSettings.json`,
  предпочитают профиль `Local.Arm.Srv` и используют его `launchUrl` плюс LDAP credentials.

Практический эффект:

- `AuthorizeFlaUiTests` больше не требуют отдельного `.runsettings`, чтобы стартовать из Visual Studio Test Explorer;
- локальный URL автоматически нормализуется в `http://localhost:5000/api/arm`;
- диагностика теперь показывает источник каждого резолвленного значения, не печатая секрет пароля;
- на 2026-03-24 полный `FlaUI`-набор (`успех/неуспех`) проходит без явных `ARM_UI_TEST_*`, когда локальный `Arm.Srv` доступен на `localhost:5000`.
